# 09 - 记忆与会话压缩

## 本文核心问题

LLM 上下文窗口有限，但对话可以无限增长。**如何在不丢失关键信息的前提下压缩历史，同时在用户无感知的情况下维护长期记忆？**

---

## 1. 三层压缩架构

压缩策略按资源消耗从低到高递进：

| 层次 | 触发方式 | 成本 | 效果 |
|------|---------|------|------|
| 微压缩（microcompact） | 时间间隔 或 缓存编辑 | 零 API 调用 | 小幅减少 |
| SM 压缩（sessionMemoryCompact） | 达到 token 阈值 + SM 功能启用 | 仅读本地文件 | 大幅减少 |
| 传统压缩（compactConversation） | SM 失败或未启用时兜底 | 一次完整 API 调用 | 最彻底 |

系统优先尝试成本低的路径，失败后降级到成本高的路径。用户执行 `/compact` 时也走相同的三条路径。

**微压缩**的触发条件是时间（距上次助手消息超过 N 分钟），不是 token 数量——这是为了在缓存即将过期前清理旧工具结果，维持 prompt cache 命中率，而不是因为上下文满了。

---

## 2. SM 压缩：解耦提取与压缩

Session Memory Compact 的精妙之处在于**把记忆提取和压缩解耦**：

- **异步提取**：每轮对话结束后，后台 forked agent 持续更新 memory file，用户无感知
- **同步压缩**：触发压缩时，直接读取已有 memory file，无需等待新 API 调用

传统压缩的问题是"压缩本身也要调用 LLM"，引入额外延迟和成本。SM 压缩把这个工作分散到每轮对话后的后台任务里，压缩触发时只需读文件，几乎是瞬时的。

如果 memory file 不存在或内容为空（比如对话刚开始），自动降级到传统压缩路径。

---

## 3. 触发条件：不是"满了才压"

**微压缩触发条件**：时间（距上次助手消息超过 N 分钟）。原因：prompt cache 有 TTL，临近过期时压缩能让下次请求重建更干净的缓存，而不是在缓存失效后仍携带大量无用的旧工具结果。

**Session Memory 提取触发条件**（两个阈值都要满足）：
1. Token 增长 >= 5000（相比上次提取）
2. 工具调用次数 >= 3，或者最后一轮是纯文本对话

第二个条件的逻辑：工具调用轮次是最有信息量的时机，每几次工具操作后提炼一次"做了什么"比每次对话都提炼更高效。纯对话轮次（没有工具调用）说明这是一个思考/总结的自然停顿点，也适合提炼。

---

## 4. API 不变量保护

Anthropic API 对消息序列有严格要求：
- `tool_result` 必须和对应的 `tool_use` 在同一个 API 轮次里
- 同一 message 的 thinking 块必须完整

压缩时如果切割点恰好把 `tool_use` 和 `tool_result` 分开，API 会报错。`adjustIndexToPreserveAPIInvariants()` 会在确定切割点后自动向前调整边界，直到满足这些约束。

这是"看起来像实现细节但实际上是正确性约束"的典型：不做这个调整，压缩后的消息历史直接无法发给 API。

---

## 5. 保留窗口的计算逻辑

SM 压缩里有一个有趣的范围计算：不是简单地"保留最近 N 条"，而是：

1. 从已摘要边界的下一条开始
2. 如果保留的内容还不够（< minTokens 或 < minTextBlockMessages），向前扩展
3. 如果扩展后超过 maxTokens，停止扩展
4. 最后用 API 不变量调整修正边界

minTokens（默认 10K）确保 Claude 有足够上下文继续工作；maxTokens（默认 40K）确保保留窗口不会太大；minTextBlockMessages（默认 5）确保至少有几条对话消息（不全是工具调用结果）。

---

## 6. 电路断路器

`autoCompact.ts` 实现了电路断路器模式：连续自动压缩失败超过 3 次，停止尝试自动压缩。

这防止在 API 故障时反复发起压缩请求（每次失败都占用时间和资源），同时不会永久禁用——计数器在新会话开始时重置，API 恢复后自动恢复。

---

## 7. 长期记忆：MEMORY.md 的角色

`~/.claude/memory/` 目录存放持久化的记忆文件，通过 `MEMORY.md` 作为索引入口。记忆有类型分类（user/feedback/project/reference），按修改时间排序，最新的在前。

每次对话开始时，相关记忆会被注入系统提示。"相关"的判断基于路径关联——当前工作目录关联的记忆优先注入。

`MEMORY.md` 有大小限制（200 行 / 25KB），防止入口文件无限增长后每次对话都要加载巨量内容。

---

## 设计原则提炼

1. **解耦提取和压缩**：后台持续提炼，压缩时直接用结果，避免关键路径上的 LLM 调用
2. **成本递进的多路径**：优先无成本路径，失败才升级
3. **触发时机考虑缓存生命周期**：微压缩触发时机是时间，不是容量
4. **切割不能破坏 API 约束**：压缩是语义操作，不是简单截断
5. **电路断路器保护系统**：失败不能无限重试
