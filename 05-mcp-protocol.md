# 05 - MCP 协议集成

## 本文核心问题

MCP（Model Context Protocol）是让 Claude Code 可扩展的机制——用户可以接入任意外部服务（数据库、搜索引擎、代码索引等）。核心问题：**如何把协议外部的工具和内置工具无缝融合，同时处理好认证、连接管理、命名冲突？**

---

## 1. 工具名的命名空间设计

MCP 工具命名格式：`mcp__serverName__toolName`

双下划线分隔有具体理由：Anthropic API 要求工具名只能包含 `[a-zA-Z0-9_-]`，且长度不超过 64 字符。用双下划线作分隔符，同时服务器名里的非法字符（点、空格等）被替换为单下划线，这样 `mcp__server__tool` 里的双下划线就能可靠地作为分隔符解析，不会和服务器名里的下划线混淆。

`normalizeNameForMCP()` 负责把服务器名里的非法字符替换掉。claude.ai 服务器名额外处理：合并连续下划线、去掉首尾下划线——因为 claude.ai 服务器名里可能有更多特殊字符。

---

## 2. 连接状态机

MCP 服务器连接是一个 5 态状态机：`pending → connected | failed | needs-auth | disabled`

TypeScript 的 discriminated union 在这里不只是类型系统设计，而是安全约束：只有 `type === 'connected'` 的连接对象才有 `client` 实例字段，其他状态的连接对象根本没有这个字段，在类型收窄后不可能对未连接的客户端发起 RPC 调用。

---

## 3. 并发连接的两个策略

启动时并行连接所有 MCP 服务器，但本地和远程用不同并发上限：
- 本地进程（stdio）：并发 3 — 避免进程爆炸
- 远程网络（SSE/HTTP/WebSocket）：并发 20 — 网络 I/O 密集

更重要的是**批处理策略的演化**：原来是"每批固定 N 个"，现在改成 `pMap`（槽位空闲即补充）。旧方案里，一批中最慢的服务器会阻塞后续批次开始；新方案里，任何一个槽位空出来就立即开始处理下一个，无需等整批完成。

---

## 4. 工具发现与缓存

`fetchToolsForClient()` 用 LRU 缓存，最多缓存 20 个服务器的工具列表。MCP 服务器的工具列表不会频繁变化，不需要每次都重新 RPC 拉取。

工具描述超过 2048 字节会被截断——这是防止 OpenAPI 自动生成的超长描述污染工具上下文的保护措施。模型看到 2048 字节已经足够理解工具用途。

从 MCP 工具的 `annotations` 字段读取 `readOnlyHint` 和 `destructiveHint`，映射到内置工具的 `isReadOnly()` 和 `isDestructive()` 接口——这让 MCP 工具能享受与内置工具相同的并发调度策略。

**Unicode 安全清洗**：从 MCP 服务器拿到工具列表后，先对所有字段做递归 Unicode 清洗，防止恶意服务器通过工具名或描述注入控制字符。

---

## 5. SSE 传输的一个细节

SSE（Server-Sent Events）是长连接，和普通 HTTP 请求不同。连接建立时的 fetch 可以加超时，但 SSE 数据流本身的 `EventSource` 不能加超时——超时 fetch 60 秒后会关闭连接，整个 SSE 通道就断了。

所以代码里 SSE 的 `requestInit`（用于建立连接的 fetch）和 `eventSourceInit`（用于 SSE 流的 fetch）分开配置，后者不设超时。这个分离看起来奇怪，但是 SSE 协议决定的必然结果。

---

## 6. URL Elicitation：暂停-恢复模式

当 MCP 服务器返回错误码 `-32042`（UrlElicitationRequired），表示需要用户在浏览器完成 OAuth 认证。系统的处理方式是：

1. 暂停工具调用
2. 显示 URL 给用户（或通过 hooks 自动处理）
3. 等待用户完成浏览器操作
4. 自动重试工具调用，最多 3 次

这是比"抛出错误让用户重新发起"更好的 UX。用户不需要手动重试，系统替他处理了中断-恢复的过程。

---

## 7. 认证缓存的写串行化

OAuth token 缓存有一个并发写问题：多个 MCP 服务器同时启动，可能同时触发 401 认证，然后同时尝试更新缓存文件，产生读-改-写竞态。

解决方法是**写操作串行化**：用 `writeChain = writeChain.then(...)` 把所有写操作排成一个 Promise 链，保证每次写入都是在上一次写入完成后进行。这是一个不需要锁的串行化方案。

---

## 8. 工具排序与 API 缓存

`assembleToolPool()` 里，内置工具排在 MCP 工具之前。这不是任意决定——Anthropic API 的 Prompt Cache 以工具列表中的某个位置为边界，内置工具定义很少变化（高缓存命中），MCP 工具可能频繁变化（低缓存命中）。把稳定的内置工具放前面，缓存边界之前的内容就能持续命中，减少成本。

---

## 设计原则提炼

1. **适配器模式**：MCP 工具通过 `MCPTool` 包装后实现相同的 `Tool` 接口，调度层完全透明
2. **双并发策略**：本地/远程按资源类型设不同并发上限
3. **命名空间防冲突**：双下划线 + 规范化，解决多服务器工具同名问题
4. **防御性解析**：Unicode 清洗、描述截断——外部输入不可信
