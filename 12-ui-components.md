# 12 - CLI UI 渲染系统

## 本文核心问题

Claude Code 的终端 UI 不是 `console.log`，而是一套基于 **React + Ink** 的完整渲染引擎。核心问题：**如何让终端 UI 达到接近 Web 应用的渲染性能和交互体验，同时处理好终端协议的碎片化？**

---

## 1. 为什么是 React

React 的 Virtual DOM 思路在终端渲染里同样有价值：状态变化只更新变化的部分，而不是重绘整个屏幕。

Ink 是 Claude Code 内嵌的定制版（非 npm 发布版），相比原版增加了：
- 双缓冲防撕裂
- DECSTBM 硬件滚动
- 字符池内存优化
- Kitty 键盘协议支持

这些扩展都是为了解决终端渲染的具体性能和兼容性问题，标准 Ink 没有针对性优化。

---

## 2. 渲染管道的 6 个阶段

每一帧渲染：
1. React Reconciler diff + 提交到虚拟 DOM
2. Yoga（WebAssembly Flexbox 引擎）计算布局
3. DOM 树 → 后缓冲区（backFrame）
4. frontFrame ↔ backFrame 原子交换
5. 计算两帧差异 → 生成最小 ANSI 写入序列
6. 写入 stdout

渲染以 40ms 节流（约 25fps）。每个流式 token 到来不直接触发渲染，而是在下一个 40ms 节点批量合并。这防止了高频状态更新（每秒几十个 token）导致的渲染抖动。

---

## 3. 双缓冲防撕裂

`frontFrame`（当前显示）和 `backFrame`（渲染中）是两块独立的屏幕缓冲区。新帧渲染完成后**原子交换**，`writeDiffToTerminal` 总是读到完整的新帧。

如果没有双缓冲，写入终端的过程中可能同时有新的渲染在修改缓冲区，用户看到的是半新半旧的混合状态。

---

## 4. 字符池：用整数 ID 代替字符串

每个屏幕单元格（Cell）不直接存储字符串，而是存储 `CharPool` 中的整数 ID。帧间对比（`diffEach`）只需比较整数，不需要字符串哈希或内存分配。

ASCII 字符走 `Int32Array` 快路径——128 个槽位直接按字符码索引，O(1) 查找，没有 Map 开销。终端内容约 80% 是 ASCII，这个优化命中率很高。

超链接也有类似的 `HyperlinkPool`：同一个 URL 在所有单元格里只存储一次，单元格只存 4 字节 ID。这减少了内存占用和比较开销，在有很多文件路径链接的输出里效果明显。

---

## 5. DECSTBM 硬件滚动

当 ScrollBox 只有 `scrollTop` 变化（内容在追加/滚动），没有其他布局变更时，不需要重写整个视口——可以发送 `DECSTBM + SU/SD` 命令，让终端硬件执行行移动。

区别：
- 软件重绘：逐行生成 ANSI 序列，写入所有可见行的内容
- 硬件滚动：发 2-3 个控制序列，终端自己移动显存里的数据

对流式输出场景（内容持续追加到底部），硬件滚动的性能差距可以是 100 倍。

`layoutShifted` 标志在每帧开始时重置，任何节点位置/尺寸变化会置为 true。如果这个标志为 false 且只有 scrollTop 变化，就走硬件滚动路径。

---

## 6. 键盘协议的兼容栈

终端键盘协议碎片化严重：老式 SSH 连接只有 VT100，现代 Ghostty/Kitty 有新协议。`parse-keypress.ts` 处理 6 种不同的转义序列格式。

协议优先级：Kitty（最现代，信息最丰富）> modifyOtherKeys > 传统转义 > ASCII

通过 `XTVERSION` 查询终端版本，选择最优协议。在同一套代码里兼容从 SSH 老终端到本地 Ghostty 的全部场景，用户不需要任何配置。

---

## 7. React Compiler 的自动 Memo

`App.tsx` 里的手动 `$[0]`, `$[1]`... 槽位代码不是手写的，是 **React Compiler 的编译输出**。

React Compiler 把 `useMemo` 模式编译成固定大小的槽位数组（`_c(9)` 分配 9 个槽），消除了运行时 `useMemo` Hook 的调用开销，同时保持精确的依赖追踪。

Provider 嵌套顺序也有考虑：变化最频繁的（`FpsMetricsProvider`）在最外层，变化最少的（`AppStateProvider`）在最内层。外层 Provider 更新不会触发内层重渲染，内层 Provider 更新也会触发外层——把高频变化放外层，减少不必要的内层重渲染。

---

## 设计原则提炼

1. **批量渲染**：40ms 节流，多个状态变化合并为一帧，避免渲染抖动
2. **双缓冲**：原子帧交换，防止写入中途的视觉撕裂
3. **整数 ID 代替字符串**：帧间对比是整数比较，内存紧凑，ASCII 走数组快路径
4. **硬件滚动**：纯追加场景利用终端硬件能力，避免全屏重绘
5. **协议兼容栈**：探测终端版本，自动选择最优键盘协议
